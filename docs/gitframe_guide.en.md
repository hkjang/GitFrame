# GitFrame CLI User Guide 🎥

GitFrame is a developer automation CLI that automatically builds and runs your web project, validates browser behavior using a defined YAML scenario, and packages the entire process into a high-quality **demo video (MP4), subtitles (SRT), thumbnail, execution report, and Playwright Trace**.

---

## 📂 Directory Structure

After running `gitframe init`, a `.gitframe` folder is created in your project root with the following structure:

```text
your-project-root/
├── .gitframe/
│   ├── gitframe.yaml      # GitFrame global execution and video rendering config
│   ├── demo.yaml          # Browser automation scenario definition for playback
│   └── assets/            # Optional folder for local intro/outro background assets
└── output/                # Output directory for all rendered artifacts
    ├── demo.mp4           # Final demo video (intro + browser recording + outro merged)
    ├── raw.webm           # Raw browser screen recording
    ├── thumbnail.png      # Video thumbnail image
    ├── result.json        # Detailed execution and test validation report
    ├── scenario.json      # JSON representation of the executed YAML scenario
    ├── report.html        # Visual HTML validation report
    ├── subtitles.srt      # Subtitle file
    ├── logs/              # Unified logs: server build, startup, rendering
    ├── screenshots/       # Screenshots captured during scenario (or at failure)
    └── trace.zip          # Playwright Trace Viewer data file
```

## 🛠️ Installation

### 1. Install npm dependencies
```bash
npm install
```

### 2. Install Playwright browser binaries
Download the browser engine (Chromium, etc.) required by Playwright locally.
```bash
npm run install-browsers
```

### 3. Install system libraries and fonts (Linux/WSL required)
A headless browser on Linux or WSL requires system libraries and fonts to render correctly.
```bash
# Install required system packages
npx playwright install-deps

# Install Korean (NanumGothic) font to prevent subtitle garbling
sudo apt-get install -y fonts-nanum
```

#### 💡 WSL / Headless Library Troubleshooting
On some WSL environments, the browser may crash due to missing shared libraries like `libnspr4.so`. GitFrame ships a portable `lib/` directory to resolve this — specify `LD_LIBRARY_PATH` at runtime:
```bash
LD_LIBRARY_PATH=./lib/usr/lib/x86_64-linux-gnu node ./dist/cli/index.js <command>
```

## ⚙️ Configuration Reference (`gitframe.yaml`)

`.gitframe/gitframe.yaml` defines your project's build/run settings and video output options.

```yaml
project:
  language: "go"               # Project type: node | python | go | java | compose | static
  install: "go mod download"   # Dependency installation command
  build: "go build -o server ./cmd/app"  # Build command
  start: "./server"            # Server startup command
  port: 8080                   # Port number to monitor
  env:                         # Additional environment variables to inject at runtime
    DATABASE_URL: "sqlite://data.db"

scenarioFile: ".gitframe/demo.yaml"  # Path to the browser automation scenario
outputDir: "output"                  # Directory for output artifacts

video:
  width: 1440    # Recording width in pixels (1440 recommended for quality)
  height: 900    # Recording height in pixels
  fps: 30        # Video frame rate

intro:
  title: "My Operations Platform"   # Intro card main title
  subtitle: "System Demo Video"     # Intro card subtitle
  duration: 3                       # Intro display duration in seconds

outro:
  title: "Thank You"                # Outro card main title
  subtitle: "Generated by GitFrame" # Outro card subtitle
  duration: 2                       # Outro display duration in seconds

captions: true   # Whether to burn subtitles (.srt) into the video
```

## 📝 Scenario File Reference (`demo.yaml`)

A scenario defines the sequence of browser actions to execute during playback.

### 1. Basic Structure
```yaml
start_url: "http://localhost:8080"  # Starting URL of your web application
steps:
  - id: step-1-login
    action: goto
    url: "/login"
    caption: "Navigating to the login page."
```

### 2. Supported Actions

| Action | Description | Required Parameters | Example |
| :--- | :--- | :--- | :--- |
| **`goto`** | Navigate to a path or absolute URL | `url` | `url: "/admin/dashboard"` |
| **`click`** | Click on an element | `selector` | `selector: "#submit-btn"` |
| **`fill`** | Type text into an input field | `selector`, `value` | `value: "myPassword"` |
| **`pause`** | Wait for a specified duration (ms) | `milliseconds` | `milliseconds: 5000` (5 sec) |
| **`assertVisible`** | Assert that an element is visible | `selector` | `selector: { role: "heading", name: "Welcome" }` |
| **`assertText`** | Assert that an element's text matches | `selector`, `text` | `text: "Success!"` |
| **`waitFor`** | Wait until an element appears | `selector` | `selector: ".loading-spinner"` |
| **`screenshot`** | Capture and save a screenshot | `name` | `name: "after-submit"` |

### 3. Selector Strategies
In addition to plain CSS selectors, GitFrame supports structured accessibility-based selectors.
GitFrame automatically picks the best matching element using this priority order:
> **Priority**: Test ID (`testId`) > Role (`role`, `name`) > Label (`label`) > Placeholder (`placeholder`) > Text (`text`) > CSS

#### Examples:
* **CSS**: `selector: "#username-input"`
* **Placeholder**:
  ```yaml
  selector:
    placeholder: "Enter your username"
  ```
* **Accessibility Role**:
  ```yaml
  selector:
    role: "button"
    name: "Submit"
  ```
* **Test ID** (when `data-testid` is present in the HTML):
  ```yaml
  selector:
    testId: "dashboard-card-1"
  ```

## 💻 CLI Commands

### 1. `gitframe init`
Creates sample config files and directory structure in the current working directory.
* **Creates**: `.gitframe/gitframe.yaml`, `.gitframe/demo.yaml`, `.gitframe/assets/`

### 2. `gitframe inspect`
Analyzes your project structure to detect the language (Node.js, Go, Python, Java, Docker Compose, etc.) and suggests recommended install/build/start commands and default ports.

### 3. `gitframe run`
Starts the local web server in the background based on `gitframe.yaml`. Performs health checks until the port is open and ready, then safely terminates child processes on CLI exit to avoid port conflicts.

### 4. `gitframe record`
Opens a headed browser and records your manual interactions (clicks, inputs, navigations) in real time, exporting them as a `.gitframe/demo.yaml` scenario file.
```bash
gitframe record --url http://localhost:8080 --output .gitframe/my-demo.yaml
```

### 5. `gitframe test`
Validates your written scenario in the browser without any video recording or rendering — ideal for fast iteration during development.
* Without `--headless`, you can watch the browser being automated in real time.

### 6. `gitframe render`
The core command. Runs the web app in an isolated workspace, records the screen, compiles subtitles, and produces the final `.mp4` video with intro/outro merged.

### 7. `gitframe create <repoUrl>`
Shallow-clones a remote Git repository and runs the entire pipeline end-to-end in a single command: `inspect` → `build & run` → `test` → `render`.
```bash
gitframe create https://github.com/example/my-project.git -o output-dir
```

### 8. `gitframe clean`
Removes leftover `gitframe-*` isolated build workspaces from the system temp directory to free up disk space.

### 9. `gitframe doctor`
Self-diagnoses whether all required dependencies (Git, Node.js, Playwright, FFmpeg, Docker) are installed and functioning, then prints a diagnostic report.

## ❓ Troubleshooting

### Q1. Korean/CJK font is garbled in the video output.
* **Cause**: The Korean font package is not installed on the Linux environment running the browser.
* **Fix**: Run the following command and re-render:
  ```bash
  sudo apt-get update && sudo apt-get install -y fonts-nanum
  ```

### Q2. Emojis appear as empty boxes (□) on the webpage.
* **Cause**: The system font does not include an emoji font library, or the browser fails to load it.
* **Fix**: Install an emoji font:
  ```bash
  sudo apt-get install -y fonts-noto-color-emoji
  ```

### Q3. FFmpeg fails with a `drawtext` error and produces a corrupted intro/outro.
* **Cause**: The installed FFmpeg binary was built without `drawtext` filter support (font engine integration).
* **Fix**: GitFrame automatically falls back to **solid color card mode** when `drawtext` is unavailable, ensuring the video is still produced without interruption. To fully enable font rendering, reinstall a complete FFmpeg package with `--enable-libfreetype`.
